CloudTrail logging

AWS CloudTrail records account activity and API calls. It is a key audit and investigation source because most AWS control plane actions appear as API events.

CloudTrail should be enabled in all accounts and regions. Multi-region trails reduce blind spots when attackers or administrators use regions that are not normally active.

Management events track control plane actions such as creating users, changing policies, modifying security groups, updating trails, and changing bucket policies. These events are essential for detecting unauthorized configuration changes.

Data events record object-level or resource-level activity for selected services, such as S3 object access and Lambda function invocation. They can be high volume, so enable them for sensitive buckets and critical resources.

Logs should be stored in a protected central location. The log bucket should have restricted write and read access, encryption, versioning, and controls that prevent deletion or tampering by ordinary administrators.

Important alerts include CloudTrail being disabled, logging configuration changes, new access keys, failed console logins, root account use, policy changes, security group exposure, KMS key deletion scheduling, and public S3 bucket changes.

CloudTrail works best with other telemetry. Combine it with CloudWatch alarms, EventBridge rules, GuardDuty findings, IAM Access Analyzer, Config rules, and centralized SIEM ingestion.

Common logging mistakes include enabling trails in only one region, storing logs in an insecure bucket, failing to alert on trail changes, ignoring data events for sensitive storage, and not retaining logs long enough for investigations.
