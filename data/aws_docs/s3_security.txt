S3 security

Amazon S3 stores objects in buckets. S3 security focuses on preventing unintended public access, protecting sensitive data, controlling write permissions, and maintaining audit visibility.

Block Public Access should be enabled for most buckets and accounts. Public access should be treated as an exception that requires a documented business reason and a review of bucket policy, ACLs, and object ownership settings.

Bucket policies should grant access to specific principals and actions. Avoid wildcard principals, broad s3:* actions, and unrestricted access to sensitive buckets. Prefer narrowly scoped permissions such as read-only access to a specific prefix when possible.

Server-side encryption should be enabled for sensitive data. S3-managed encryption is simple to operate, while KMS-managed keys provide stronger control, audit logs for key usage, and separation of duties.

Versioning helps recover from accidental deletion, overwrite, and some ransomware-style events. For critical buckets, combine versioning with lifecycle rules, object lock where appropriate, and tested recovery procedures.

Logging and monitoring should cover data access and configuration changes. CloudTrail data events can record object-level access for important buckets. Alerts should watch for public policy changes, disabled encryption, unusual download volume, and access from unexpected principals.

Sensitive buckets should use clear naming, tags, ownership, and classification. Data owners should know what the bucket stores, who can access it, and whether it contains regulated or confidential information.

Common S3 risks include public read access, public write access, overly broad bucket policies, disabled encryption, missing logging, anonymous uploads, exposed static website buckets, and cross-account access that is not reviewed.
