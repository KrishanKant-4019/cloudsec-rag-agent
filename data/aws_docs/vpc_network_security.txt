VPC and network security

Cloud network security controls which systems can communicate, which workloads are exposed to the internet, and how traffic is inspected or logged. In AWS, VPCs, subnets, route tables, security groups, network ACLs, gateways, and endpoints are core controls.

Public subnets should contain only resources that must receive direct internet traffic, such as load balancers or bastion services. Databases, internal APIs, message brokers, and administrative systems should normally live in private subnets.

Security groups act as stateful virtual firewalls. Inbound rules should be narrow: restrict ports, protocols, and source ranges. Avoid 0.0.0.0/0 for administrative ports such as SSH, RDP, database ports, and management consoles.

Network ACLs provide subnet-level controls and can help with coarse filtering, but security groups are usually easier to manage for workload-level access. Use both intentionally instead of relying on accidental defaults.

VPC endpoints can keep traffic to supported AWS services off the public internet. They also allow endpoint policies to limit which services or resources can be reached from a network path.

Egress should be considered, not only ingress. Unrestricted outbound traffic can allow malware command-and-control, data exfiltration, or access to unapproved external services. Sensitive workloads may need restricted egress through inspection points.

Network visibility matters. VPC Flow Logs, load balancer logs, DNS logs, and firewall logs help identify scanning, unexpected access, blocked traffic, and unusual outbound connections.

Common network risks include public databases, wide-open administrative ports, unused public IPs, overly permissive security groups, flat network design, missing flow logs, and lack of separation between production and development systems.
