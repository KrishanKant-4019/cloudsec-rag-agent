IAM security

Identity and Access Management controls authentication and authorization for cloud resources. In AWS, IAM users, groups, roles, policies, and permissions determine who can perform actions against services and resources.

Least privilege is the central IAM principle. A principal should receive only the permissions required for its task, for only the resources it needs, and only for the necessary time. Avoid broad actions such as "*" and broad resources such as "*" unless there is a clear and reviewed reason.

Roles are preferred for workloads and temporary access because they avoid long-lived static credentials. Applications running on cloud services should use service roles or instance profiles instead of embedded access keys.

Multi-factor authentication should be required for human users, especially administrators and users who can change identity, billing, logging, network, or security controls.

Privileged access should be separated from daily work. Administrators should use separate elevated roles, approval workflows, session logging, and short session durations.

IAM policies should be reviewed for risky permissions. High-risk patterns include iam:PassRole with broad resources, sts:AssumeRole into sensitive roles, wildcard administrative actions, permissions to edit logging, and access to secrets or encryption keys.

Access keys should be minimized. If long-lived keys are unavoidable, rotate them, monitor their use, and remove unused keys. Never store keys in source code, public repositories, AMIs, container images, or unencrypted configuration files.

Good IAM operations include regular access reviews, removing unused users and roles, using permission boundaries where appropriate, monitoring failed authorization attempts, and alerting on policy changes or new administrator access.
