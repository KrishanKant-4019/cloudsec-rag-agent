Common cloud misconfigurations

Cloud misconfigurations are insecure settings that expose data, grant excessive access, disable visibility, or weaken resilience. They are common because cloud environments are API-driven, fast changing, and often managed by many teams.

Public storage is a frequent risk. Buckets or objects may become readable or writable by anyone due to broad policies, ACLs, static website settings, or accidental cross-account access.

Overly permissive IAM creates privilege escalation and blast-radius problems. Risky examples include administrator policies on routine users, wildcard actions, wildcard resources, broad iam:PassRole, and long-lived access keys.

Network exposure can make private systems reachable from the internet. Examples include databases with public IP addresses, security groups open to 0.0.0.0/0, unrestricted SSH or RDP, and public load balancers pointing to sensitive applications.

Missing logging reduces detection and investigation ability. CloudTrail disabled, incomplete regional logging, missing data events, unprotected log buckets, and short retention periods all create blind spots.

Weak encryption settings can expose sensitive data. Examples include unencrypted storage, public snapshots, weak TLS policy, excessive KMS decrypt permissions, and unreviewed key deletion.

Secrets exposure is another major pattern. Secrets may appear in code repositories, CI logs, container images, user data scripts, environment variables, or shared documents.

Good prevention includes infrastructure-as-code scanning, policy-as-code guardrails, secure defaults, continuous configuration monitoring, access reviews, alerting on risky changes, and regular cleanup of unused resources.
