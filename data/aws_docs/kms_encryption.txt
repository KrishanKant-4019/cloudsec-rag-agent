KMS and encryption

Encryption protects data confidentiality by making stored or transmitted data unreadable without the right keys. In AWS, Key Management Service helps create, protect, rotate, and audit cryptographic keys used by many cloud services.

Encryption at rest protects stored data such as S3 objects, database records, EBS volumes, backups, and logs. Encryption in transit protects network communication using protocols such as TLS.

AWS-managed keys are easy to use and require less administration. Customer-managed KMS keys provide more control over policies, rotation, separation of duties, and auditing.

KMS key policies are security-critical. A key policy should define who can administer the key, who can use it for encryption or decryption, and which services or accounts are allowed. Avoid broad key administrators and broad decrypt access.

Separation of duties is important. Users who can administer a key should not automatically be able to decrypt all protected data unless that access is explicitly required.

KMS usage appears in audit logs. Monitoring decrypt operations, key policy changes, disabled keys, scheduled deletion, and unusual cross-account usage can help detect misuse.

Key deletion should be controlled carefully. Scheduling deletion for a KMS key can make protected data unrecoverable after the waiting period. Critical keys need ownership, recovery plans, and change approval.

Common encryption risks include storing sensitive data unencrypted, allowing too many principals to decrypt, using unmanaged keys without rotation plans, missing TLS enforcement, and failing to monitor key policy changes.
