Secrets management

Secrets are sensitive values such as passwords, API keys, tokens, private keys, database credentials, and webhook signing keys. Poor secrets handling is a common cause of cloud compromise.

Secrets should not be stored in source code, public repositories, container images, AMIs, logs, build output, or plaintext configuration files. They should be stored in a managed secrets service or secure parameter store with encryption and access control.

Applications should retrieve secrets at runtime using an assigned role or managed identity. This avoids distributing long-lived credentials to developers, build systems, or deployment scripts.

Access to secrets should follow least privilege. A workload should read only the secrets it needs. Human access to production secrets should be rare, approved, logged, and time limited.

Secret rotation reduces the impact of exposure. Rotation should be tested so applications can handle updated credentials without outages. Emergency rotation procedures should exist for leaked keys.

Monitoring should detect suspicious secret access. Alerts should watch for unusual principals reading secrets, sudden spikes in reads, cross-account access, and changes to secret resource policies.

Environment variables can be convenient but require caution. They may leak through debug output, process inspection, crash reports, or misconfigured CI logs. Managed secret injection is usually safer for production.

Common secrets risks include committed API keys, over-permissive secret policies, no rotation, shared credentials, secrets in logs, unencrypted configuration, and old credentials that remain active after migration.
